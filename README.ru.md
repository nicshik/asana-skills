# Asana Skills

[English](README.md)

Codex-навыки и небольшие Python-скрипты для работы напрямую с REST API Asana.

Репозиторий полезен, когда стандартный коннектор Asana доступен только для чтения, недоступен или заблокирован защитой инструментов, но нужен узкий и проверяемый способ автоматизировать действия в Asana собственным персональным токеном.

Репозиторий намеренно generic. Конкретные воркспейсы, проекты, теги, исполнители и правила "готово" должны жить в отдельной обёртке-навыке или проектном репозитории, а не здесь.

## Состав навыков

| Навык | Назначение |
| --- | --- |
| `asana-preflight` | Проверка токена: возвращает пользователя и воркспейсы. |
| `asana-read-task` | Чтение одной задачи (опционально со stories) как read-only fallback. |
| `asana-list-tasks` | Список задач по проекту, секции, тегу или исполнителю, с пагинацией. |
| `asana-search-tasks` | Поиск задач в воркспейсе (премиум-функция Asana). |
| `asana-create-task` | Создание задачи с контейнером, секцией, родителем, исполнителем и тегами. |
| `asana-update-task` | Обновление полей, статуса завершения, тегов или секции задачи. |
| `asana-delete-task` | Удаление задачи после чтения, с dry-run и точным подтверждением. |
| `asana-comment-task` | Добавление одного комментария (story) к задаче. |
| `asana-list-projects` | Список проектов в воркспейсе или команде. |
| `asana-project-sections` | Список секций проекта или создание одной секции. |
| `asana-list-tags` | Список тегов воркспейса. |
| `asana-api` | Универсальный passthrough к любому эндпоинту REST API Asana. |

## Требования

- Python 3.10 или новее.
- Персональный токен Asana с доступом к нужному воркспейсу.
- Желательно: `certifi` для надёжной проверки TLS-сертификатов.

```bash
python3 -m pip install -r requirements.txt
```

## Токен

Создайте токен на `https://app.asana.com/0/my-apps` ("Personal access tokens") и задайте его:

```bash
export ASANA_ACCESS_TOKEN=<asana-personal-access-token>
```

Или передайте локальный env-файл (`--env-file`). Содержимое:

```text
ASANA_ACCESS_TOKEN=<asana-personal-access-token>
```

Не коммитьте реальные токены: `.env` и `.env.*` игнорируются. Токен даёт широкий доступ к аккаунту — держите операции в пределах нужного воркспейса и проекта.

## Использование

Все gid ниже — заглушки.

```bash
python3 asana-preflight/scripts/preflight.py --env-file /path/to/.env.local --json
python3 asana-read-task/scripts/read_task.py 1200000000000000 --env-file /path/to/.env.local --include-stories
python3 asana-create-task/scripts/create_task.py --project 1200000000000001 --name "Example task" --notes-file /path/to/body.txt --env-file /path/to/.env.local --dry-run --json
```

`--json` — когда вывод потребляет другой инструмент или агент. `--dry-run` — для проверки полезной нагрузки перед записью.

## Обёртки под проект

Это низкоуровневые помощники. Они не решают, какую задачу создавать, какой воркспейс/проект верный и завершена ли работа. Эти решения держите в отдельной обёртке-навыке, которая вызывает скрипты через `ASANA_ACCESS_TOKEN`, `ASANA_ENV_FILE` или `--env-file` и передаёт собственные конкретные gid.

## Модель безопасности

- Токен читается только из переменных окружения или локальных env-файлов; в выводе никогда не печатается и редактируется в ошибках.
- Жёсткий таймаут запроса не даёт вызовам зависать.
- Read-only навыки не отправляют записей; `--dry-run` на пишущих навыках резолвит полезную нагрузку без мутаций.
- `scripts/validate.sh` блокирует случайные project-specific, account- или local-machine строки, чтобы публичный репозиторий оставался переносимым.

## Разработка

```bash
scripts/validate.sh
python3 -m unittest discover -s tests
scripts/secret_scan.sh
```

CI не вызывает живой API Asana и не требует токена; тесты используют моки и фикстуры.
